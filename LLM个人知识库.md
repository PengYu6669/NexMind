一、原架构核心设计理念（先定总纲，发给 Cursor 先对齐思想）
核心原则：LLM 是知识库第一维护者，人只做浏览 / 输入，不做手工整理
所有外部资料统一进入 raw/原始资料池（原始、不修改、不整理、原汁原味）
LLM 全自动对 raw 内容做消化处理，结构化写入 wiki/.MD结构化知识库
wiki 知识库再衍生产出问答、笔记、图表等，存入 outputs/派生结果池
输出结果、查询问答、健康检查，会回写更新 wiki 知识库，实现知识库越用越厚、持续自增长，而非越查越散
Obsidian/Web 前端仅作为浏览界面，不参与内容生产维护
二、原版完整全链路流程（1:1 还原你的图，标准化分步）
阶段 1：摄取 Ingestion → 全部资料汇入 raw / 原始资料池
负责：把一切外部非结构化信息，统一转为原始文本资料，存入 raw，不做任何 AI 消化整理现有 3 个摄取入口：
网页、论文、代码仓库数据集 → Web Clipper 抓取正文 → 原始文本存入raw/
播客、音频文件 → 语音转录逐字稿 → 原始文稿存入raw/
手动剪藏、截图、本地图片文件 → 手动录入原文 / 标注 → 原始资料存入raw/
原版现存致命缺陷：视频文件 / 在线视频链接无法摄取豆包原生 API 无法直接解析视频画面 / 音频，视频无法转为原始文本进入 raw，整个后续消化、wiki 链路完全中断。
阶段 2：消化 Digestion → LLM 全自动把 raw 转为 wiki 结构化知识库
LLM 持续后台异步执行，完全自动化，无需人工干预，3 个子任务：
摘要提取：对每一份 raw 原始资料，生成精简核心摘要 summary
概念 & 主题文章：拆解知识点，生成独立概念页、主题结构化长文 concepts/articles
索引、回链、分类：全局构建知识库索引、双向回链、分类目录、知识结构 index/backlinks/structure
全部处理完成后，统一写入 wiki/.MD结构化知识库
阶段 3：输出 Output → wiki 知识库衍生产出，存入 outputs / 派生结果池
基于 wiki 结构化知识库，自动生成对外可用成果：
问答归档 q&a archive：用户查询问答、历史对话沉淀归档
Markdown 笔记、幻灯片、Matplotlib 图表 notes/slides/matplotlib
健康检查 & 缺失值补齐 consistency/missing data：检测知识冲突、缺失、过期，自动修复补齐
全部结果存入 outputs/派生结果池。
阶段 4：闭环自增长 Query Loop（原图底部核心闭环，原版遗漏重点）
用户查询知识库 → LLM 先读 wiki 索引摘要 → 检索对应知识 → 回答用户问题
复杂问答结果、健康检查修复内容，自动回写更新 wiki 知识库
知识库持续迭代扩容，形成正向自增长循环

一、系统总目标
搭建自研 Web 端、LLM 全自动驱动、自增长迭代的个人知识库系统，遵循 摄取raw → LLM消化wiki → 输出outputs → 回写wiki自增长 三段式闭环架构；原生已支持网页 / 论文 / 音频 / 手动资料摄取，本次新增视频全链路摄取能力，补齐原有视频无法被 LLM 学习的短板，全程复用现有消化、输出、回写逻辑，不改动原有业务架构。
二、目录结构定义（固定不变，和原图完全一致）
raw/：原始资料池，仅存储未经 LLM 结构化处理的原始原文资料，只读原始数据，不修改
wiki/：.MD 结构化知识库，LLM 唯一维护的核心知识数据库
outputs/：派生结果池，存储问答、笔记、图表、健康检查等衍生产出

完整摄取全入口：

手动剪藏、截图配图、本地文件
